# CLAUDE.md

运用第一性原理思考，拒绝盲从既有经验或陈旧的路径。在执行任何编码、重构或设计任务前，请严格遵循以下思维链路：

1. **溯源本质**：不要假设当前方案是最优解。从最原始的业务需求和技术问题出发，剥离表象，直击核心目标。
2. **审慎假设**：若目标不够清晰，请立即停止执行并与我讨论。不要预设你完全清楚我想要的最终结果。
3. **路径优化**：即便目标清晰，也请评估当前拟定的路径是否最高效。如果有更短、更低成本或更优雅的替代方案（防止“XY问题”），请优先向我提出建议。

## 执行规范
当接收到任务时，请以以下结构组织你的回复：
- 🟢 **第一性分析**：用 1-2 句话拆解当前任务的本质目标，并评估是否有更优的替代路径。
- ⚡ **直接执行**：按当前要求直接给出任务的执行步骤或代码。

## 安全与错误处理

* 所有用户输入必须校验
* 所有敏感接口必须校验权限
* 不允许信任前端传来的数据
* 不允许把密钥、Token、数据库密码写进代码
* 不允许把数据库错误、服务器路径、密钥等敏感信息返回给前端
* 后端必须有统一错误处理
* 不允许 catch 后静默失败
* 不允许返回假成功

## 代码阅读与文档优先

阅读或修改代码前，必须遵循以下顺序：

1. **先读文档**：优先阅读 `docs/` 目录下的相关文档（如 `architecture.md`、`components.md`、`api.md` 等），建立全局认知
2. **再读代码**：基于文档理解，再定位和阅读具体源代码
3. **文档同步**：修改任何模块后，必须同步更新对应的 docs 文档，确保文档与代码一致

## 修改代码前必须先做

每次修改前，必须先说明

1. 当前目录结构判断
2. 本次涉及前端、后端、数据库还是共享类型
3. 会新增哪些文件
4. 会修改哪些文件
5. 每个文件的职责
6. 是否复用已有模块，还是新增模块

说明完成后，再开始写代码

## 严格禁止

* 禁止把完整功能写进一个文件
* 禁止把所有接口写进一个文件
* 禁止把所有样式写进一个文件
* 禁止在组件中同时堆 UI、状态、请求、复杂计算
* 禁止在 controller 中写复杂业务逻辑
* 禁止在 repository 中写业务规则
* 禁止在没有理解现有结构前直接大改
* 禁止重复造组件、hook、service
* 禁止引入不必要的复杂依赖
* 禁止为了快而牺牲结构
* 禁止吞掉错误
* 禁止绕过参数校验和权限校验

## 完成后必须输出

每次完成任务后，必须说明

* 新增了哪些文件
* 修改了哪些文件
* 删除了哪些文件
* 每个文件的职责
* 前端改动点
* 后端改动点
* API 是否变化
* 数据库是否变化
* 类型是否变化
* 是否需要补充测试
* docs 文档是否已同步更新
* 是否存在可继续拆分或优化的地方
